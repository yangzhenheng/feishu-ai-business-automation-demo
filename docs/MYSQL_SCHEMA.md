# MySQL Schema 设计说明

本项目的 MySQL 设计用于展示女装电商在飞书低代码场景下如何承载 ERP / WMS / CRM / AI 日报数据。它不是生产级完整建模，而是面向求职展示和面试演示的业务闭环样例。

## 核心表

| 表名 | 业务归属 | 作用 |
|---|---|---|
| `orders` | ERP | 承载店铺订单、SKU、金额、买家备注和订单状态，是业务闭环入口。 |
| `inventory` | WMS | 承载仓库、可用库存、锁定库存和安全库存，用于低库存预警。 |
| `exceptions` | 协同流程 | 承载库存不足、物流延迟、退换货、少发漏发等异常。 |
| `customer_leads` | CRM | 承载直播间、店铺客服、售后沉淀的客户线索。 |
| `ai_reports` | AI 日报 | 保存每日经营摘要、风险数量、使用模型和是否推送飞书。 |
| `workflow_logs` | 飞书自动化 | 记录审批流、任务创建、Webhook 推送等自动化动作。 |
| `finance_reconcile` | 财务 | 承载订单金额、实收金额、差异金额和差异原因。 |
| `logistics_tracking` | 物流 | 承载运单状态、异常原因和建议动作。 |

## 如何支撑 ERP / WMS / CRM

`orders` 是 ERP 订单池，店铺订单同步后统一写入该表。运营、客服和仓储都围绕订单状态协同处理。

`inventory` 对应 WMS 库存，按 SKU 和仓库管理可用库存、安全库存和补货周期。当可用库存低于安全库存时，可以触发采购补货审批。

`customer_leads` 对应 CRM 线索池，把直播间团购、店铺咨询、售后工单中的高价值客户沉淀出来，并分配推荐负责人。

`finance_reconcile` 和 `logistics_tracking` 让订单闭环不止停留在“已发货”，而是继续覆盖回款、对账、物流履约和售后风险。

## 如何支撑 AI 日报

AI 日报可以从这些表聚合出：

- 今日订单数、成交金额、异常数量。
- 低库存 SKU 和缺口数量。
- 未关闭 P0/P1 异常和责任人。
- 物流延迟、退换货、少发漏发风险。
- 财务对账差异及原因。
- 高意向客户线索和下一步跟进动作。

`ai_reports` 保存 AI 生成结果，`workflow_logs` 记录日报推送、审批创建和异常升级动作，形成可追踪的业务闭环。
